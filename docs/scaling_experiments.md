# Scaling Experiments

This guide explains the scaling part of the experiment code: which datasets are
used, which scripts run the scaling batches, and how to read the generated CSV
and plot outputs.

For normal use, start with the Windows profile runner described in
[README.md](../README.md). The runner calls the scaling scripts and places their
outputs under one timestamped run folder.

## What Scaling Measures

Scaling experiments run basis, amplitude, and phase encoding workflows over
synthetic and local real knowledge graph inputs.

They measure:

- RDF parsing time
- ID mapping time
- state preparation time
- circuit construction time
- simulation time
- measurement time
- qubit counts
- logical circuit depth and gate count
- optional decomposed/transpiled circuit metrics
- status for successful, skipped, limited, timed-out, or failed rows

The scaling scripts produce software measurements from local simulation. They do
not produce hardware results.

## Recommended Entry Point

Use:

```powershell
.\scripts\run_experiments_windows.ps1 -Profile light -ResultsRoot results\runs
.\scripts\run_experiments_windows.ps1 -Profile medium -ResultsRoot results\runs
.\scripts\run_experiments_windows.ps1 -Profile hard -ResultsRoot results\runs
```

Scaling outputs from those runs appear under:

```text
results/runs/<timestamp>_<profile>/raw/scaling/
```

The collector also folds scaling metrics into:

```text
results/runs/<timestamp>_<profile>/summary/results_summary.csv
```

## Synthetic Datasets

Synthetic datasets are deterministic Turtle files generated by:

```powershell
python scripts/generate_scaling_datasets.py
```

Default generated files:

| File | Triples | Entities | Predicates |
| --- | ---: | ---: | ---: |
| `data/scaling/synthetic_100.ttl` | 100 | 50 | 5 |
| `data/scaling/synthetic_1000.ttl` | 1,000 | 500 | 10 |
| `data/scaling/synthetic_5000.ttl` | 5,000 | 2,500 | 15 |
| `data/scaling/synthetic_10000.ttl` | 10,000 | 5,000 | 20 |

The files use simple triples like:

```turtle
@prefix ex: <http://example.org/> .

ex:e0 ex:p0 ex:e1 .
ex:e1 ex:p1 ex:e2 .
```

To add a size, edit `DATASET_CONFIGS` in
`scripts/generate_scaling_datasets.py`, then regenerate.

## Local Real Datasets

Local real KG files are expected in:

```text
data/real_kgs/
```

Current default files:

```text
data/real_kgs/Aristotle.xml
data/real_kgs/DecodedOntologies_V2.ttl
data/real_kgs/exampleV3.ttl
data/real_kgs/productsSmall.rdf
```

The scripts do not download these files. If a file is missing or cannot be
parsed, the row is recorded with an error status and the batch continues when
the profile allows it.

Supported suffixes include:

- `.ttl`
- `.rdf`
- `.xml`
- `.nt`

## Direct Scaling Commands

Use direct commands when you are developing or debugging scaling behavior.

Synthetic only:

```powershell
python scripts/run_scaling_experiments.py --sizes 100 1000 --repetitions 1 --shots 1000 --output-dir results\scratch_scaling
```

Synthetic plus local real data:

```powershell
python scripts/run_all_experiments.py --results-root results\scratch_all --synthetic-sizes 100 1000 --repetitions 1 --shots 1000 --timeout-seconds 300
```

Synthetic only through the all-in-one Python script:

```powershell
python scripts/run_all_experiments.py --results-root results\scratch_all --skip-real --synthetic-sizes 100 1000 --repetitions 1 --shots 1000 --timeout-seconds 300
```

Real only:

```powershell
python scripts/run_all_experiments.py --results-root results\scratch_real --skip-synthetic --real-files exampleV3.ttl productsSmall.rdf --repetitions 1 --shots 1000 --timeout-seconds 300
```

Append new scaling rows to an existing output directory:

```powershell
python scripts/run_scaling_experiments.py --sizes 100 1000 --repetitions 1 --output-dir results\scratch_scaling
python scripts/run_scaling_experiments.py --sizes 5000 --repetitions 1 --append --output-dir results\scratch_scaling
```

Regenerate summaries and plots from existing raw rows:

```powershell
python scripts/run_scaling_experiments.py --regenerate-plots-only --output-dir results\scratch_scaling
```

When called by the Windows runner:

```text
results/runs/<timestamp>_<profile>/raw/scaling/
|-- synthetic/
|   |-- scaling_raw_results.csv
|   |-- scaling_summary.csv
|   `-- plots/
|-- real/
|   |-- scaling_raw_results.csv
|   |-- scaling_summary.csv
|   `-- plots/
`-- combined/
    |-- scaling_raw_results.csv
    |-- scaling_summary.csv
    `-- plots/
```

When run directly with `--output-dir` or `--results-root`, outputs are written
under the directory you provide.

## Raw CSV Columns

`scaling_raw_results.csv` keeps every attempted row. Important columns include:

- `dataset_category`
- `dataset_name`
- `dataset_path`
- `dataset_size`
- `num_triples`
- `num_entities`
- `num_predicates`
- `encoding`
- `repetition`
- `success`
- `status`
- `error_message`
- `total_runtime`
- `rdf_parse_time`
- `id_mapping_time`
- `state_preparation_time`
- `circuit_construction_time`
- `simulation_time`
- `measurement_time`
- `qubits`
- `logical_circuit_depth`
- `logical_gate_count`
- `decomposed_circuit_depth`
- `decomposed_gate_count`
- `transpiled_circuit_depth`
- `transpiled_gate_count`
- `metric_status`
- `metric_error`
- `shots`
- `backend`
- `timeout_seconds`
- `max_real_triples`

`scaling_summary.csv` groups rows by dataset and encoding and reports run
counts, success/failure counts, mean timings, and mean circuit metrics.

## Status Values

| Status | Meaning |
| --- | --- |
| `success` | The row completed. |
| `simulator_limit` | A configured dense simulation guard stopped the row before it became unsafe. |
| `timeout` | The row exceeded the configured timeout. |
| `parse_error` | RDF parsing failed for that dataset. |
| `error` | Another exception occurred. See `error_message`. |
| `skipped` | A row was intentionally skipped, usually because an optional group was not enabled. |

Failures and limits are part of the result. Do not replace them with guessed
numbers.

## Circuit Metric Levels

The scripts record multiple circuit metric levels:

| Metric family | Meaning |
| --- | --- |
| `logical_*` | Metrics on the original high-level circuit. |
| `decomposed_*` | Metrics after Qiskit decomposition, when enabled and within guardrails. |
| `transpiled_*` | Metrics after Qiskit transpilation, when enabled and within guardrails. |

Metric extraction can become expensive. Use these flags to control it:

```powershell
--max-metric-qubits 14
--compute-decomposed-metrics
--no-compute-decomposed-metrics
--compute-transpiled-metrics
--no-compute-transpiled-metrics
```

If a metric is not available, check `metric_status` and `metric_error`.

## Simulation Guardrails

Useful guardrails:

```powershell
--max-basis-simulation-qubits 22
--max-phase-diagonal-qubits 10
--max-metric-qubits 14
--timeout-seconds 300
```

Basis encoding can require many qubits because it encodes:

```text
subject_bits || predicate_bits || object_bits
```

Phase encoding currently uses a dense diagonal path for some operations, so it
also has a qubit guardrail. Amplitude encoding often uses fewer qubits, but
state preparation and simulation can still dominate runtime.

## Phase Predicate Selection

Phase scaling needs a rule for selecting marked triples. Available modes:

| Mode | Behavior |
| --- | --- |
| `synthetic-default` | Mark `http://example.org/p0`. |
| `first-predicate` | Mark the first predicate in the parsed deterministic triple list. |
| `most-common-predicate` | Mark the most frequent predicate. |
| `none` | Mark no triples. |
| `custom` | Mark the predicate supplied by `--phase-mark-predicate`. |

Useful columns:

- `phase_marker_mode`
- `phase_requested_predicate`
- `phase_effective_predicate`
- `phase_marked_triples`
- `phase_total_triples`
- `phase_marked_fraction`
- `phase_warning`

If `phase_marked_triples` is zero, the row is still valid audit data, but it is
a no-op phase run.

## Plot Files

Common plot names include:

- `<group>_total_runtime_vs_triples.png`
- `<group>_state_preparation_time_vs_triples.png`
- `<group>_simulation_time_vs_triples.png`
- `<group>_qubits_vs_triples.png`
- `<group>_logical_circuit_depth_vs_triples.png`
- `<group>_logical_gate_count_vs_triples.png`
- `<group>_decomposed_circuit_depth_vs_triples.png`
- `<group>_decomposed_gate_count_vs_triples.png`
- `<group>_transpiled_circuit_depth_vs_triples.png`
- `<group>_transpiled_gate_count_vs_triples.png`

For real dataset bar charts, filenames use `_by_dataset` instead of
`_vs_triples`.

Rows with failure, timeout, or simulator-limit statuses remain in the raw CSV.
They are not converted into fake plot points.

## Reading Scaling Results In A Runner Folder

Start here:

```text
run_manifest.json
summary/results_summary.csv
raw/scaling/combined/scaling_summary.csv
raw/scaling/combined/scaling_raw_results.csv
raw/scaling/combined/plots/
logs/scaling_experiments.log
```

Use `summary/results_summary.csv` when you want one normalized metrics table
across the whole run. Use the raw scaling CSV when you need row-level status and
error details.
